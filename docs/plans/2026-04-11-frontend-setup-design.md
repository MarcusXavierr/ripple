# Frontend Setup Design — Ripple

## Context

Ripple is a browser-based P2P video call app. The `ripple/` directory is the frontend root. Backend lives in a separate repo. This document covers the initial project setup only.

## Stack

| Concern | Choice |
|---|---|
| Runtime | Bun |
| Framework | Vite + React + TypeScript |
| Styling | Tailwind CSS + shadcn/ui |
| State | Zustand |
| Routing | React Router |
| Lint + Format | Biome |
| Tests | Vitest + React Testing Library + jsdom |
| Test style | TDD, test files colocated with source |

## Project Initialization

Scaffold Vite into the existing repo root (no subdirectory):

```bash
bun create vite . --template react-ts
```

## Tooling

### Biome

Single tool replacing both ESLint and Prettier. Config lives at `biome.json` in the project root.

Scripts:
- `bun run lint` — lint only
- `bun run format` — format only
- `bun run check` — lint + format together (used in CI / pre-commit)

### Vitest + RTL

Configured inside `vite.config.ts` — no separate config file. Test environment: `jsdom`.

A `vitest.setup.ts` at root imports `@testing-library/jest-dom` to enable custom matchers (`.toBeInTheDocument()`, etc.).

Scripts:
- `bun run test` — watch mode during development
- `bun run test:ui` — Vitest UI

## Dependencies

```bash
# Runtime
bun add react-router-dom zustand

# Tailwind (via Vite plugin)
bun add -d tailwindcss @tailwindcss/vite

# shadcn/ui (initialized after Tailwind — adds components on demand)
bunx shadcn@latest init

# Vitest + RTL
bun add -d vitest @vitest/ui jsdom
bun add -d @testing-library/react @testing-library/user-event @testing-library/jest-dom
```

## Folder Structure

```
src/
├── main.tsx
├── router.tsx
├── types/
│   └── signaling.ts        # SYNC: keep identical to backend types.ts
├── store/
│   ├── call.ts             # Zustand store
│   └── call.test.ts
├── hooks/
│   ├── useWebRTC.ts
│   └── useWebRTC.test.ts   # written when hook is implemented
└── pages/
    ├── Home.tsx
    ├── Home.test.tsx
    ├── Room.tsx
    └── Room.test.tsx
```

shadcn/ui components land in `src/components/ui/` (generated by the CLI).

## TDD at Setup

Tests written during setup — before any real logic:

### Smoke tests (one per page)

```tsx
// Home.test.tsx
it('renders without crashing', () => {
  render(<Home />)
})

// Room.test.tsx — needs mocked router params
it('renders without crashing', () => {
  render(
    <MemoryRouter initialEntries={['/room/test-room']}>
      <Routes><Route path="/room/:id" element={<Room />} /></Routes>
    </MemoryRouter>
  )
})
```

### Store initial state

```ts
// call.test.ts
it('initializes with correct default state', () => {
  const state = useCallStore.getState()
  expect(state.status).toBe('idle')
  expect(state.ws).toBeNull()
  expect(state.pc).toBeNull()
  expect(state.role).toBeNull()
  expect(state.peerId).toMatch(/^[0-9a-f-]{36}$/)  // valid UUID
})
```

### Nothing else at setup

- `signaling.ts` — types only, no runtime, no test
- `useWebRTC.ts` — tests written when the hook is implemented
- `router.tsx` — covered by the page smoke tests

## What is NOT in scope for setup

- Any WebRTC logic
- Any WebSocket logic
- UI implementation
- shadcn/ui component installation (done on demand per feature)
